## Summary

The implementation ships **all ten features with correct core business logic** and passes the security gate. However, **zero automated test files exist** for any feature in the project (all PRDs specify `.test.ts` / `.spec.ts` files that were never created), which blocks CI verification entirely — this is a hard blocker. Three additional Critical issues were found: a hardcoded `0` bug in `wikiExport.ts`'s `ExportResult`, a broken diff UI in the PR bot wizard, and the complete absence of the F007 `claude-plugin` slash-command template directory. Four Warnings and four Suggestions complete the report. Recommendation: **do not ship until the 4 Criticals are resolved**; Warnings should be addressed before the next release.

  **Remediation:** Create the test files specified in each PRD's "New files" section. For F004 `cronParser.test.ts` this is especially urgent — the PRD documents 30 specific test cases covering all field types, edge dates, and `isOverdue` logic.

- [ ] **F003: `ExportResult.wikilinksResolved` hardcoded to `0`** — `src/driver/wikiExport.ts:71`

  The `exportWiki` function correctly tracks resolved wikilinks inside `exportStaticHtml` and `exportDocusaurus` via local variables, but the `return` statement on line 71 ignores those variables and always returns `0`:

  ```typescript
  // wikiExport.ts:67-73
  let wikilinksResolved = 0;
  if (target === "html") {
    wikilinksResolved = await exportStaticHtml(pages, wikiDir, wikiIndex, tmpDir);
  } else if (target === "docusaurus") {
    wikilinksResolved = await exportDocusaurus(pages, wikiDir, wikiIndex, tmpDir, projectName);
  }
  // ...
  return {
    pagesExported: pages.length,
    wikilinksResolved: 0,   // ← hardcoded; should be `wikilinksResolved`
    outputDir,
    durationMs: Date.now() - startMs,
  };
  ```

  AC-3.1.3 (wikilink conversion) and the PRD's success metric "Wikilink resolution accuracy ≥ 98%" require that the caller can observe how many links were resolved. The MCP tool (future `legion_export_wiki`) would expose this count to agents. Callers currently receive a misleading `0` regardless of actual link resolution.

  **Remediation:** Replace `wikilinksResolved: 0` with `wikilinksResolved` (the tracked variable) on `wikiExport.ts:71`.

- [ ] **F006: "Show diff" in PR bot wizard displays blank right pane** — `src/commands/installPrReviewBot.ts:79`

  When the user selects "Show diff" for a modified workflow file, the implementation opens a diff editor with an `untitled:` URI for the right pane:

  ```typescript
  // installPrReviewBot.ts:75-82
  await vscode.commands.executeCommand(
    "vscode.diff",
    vscode.Uri.file(workflowPath),
    vscode.Uri.parse(`untitled:legion-wiki-diff.yml (latest template)`),  // ← empty document
    "legion-wiki-diff.yml (existing ↔ latest)"
  );
  ```

  VS Code's `untitled:` URI creates a new empty document, not a document populated with the template content. The PRD specifies a `data:text/plain,${encodeURIComponent(bundledTemplate)}` URI to encode the template inline. The result is that users see "existing file" vs "empty document" — they cannot assess what would change if they overwrite.

  **Remediation:** Replace `vscode.Uri.parse('untitled:...')` with `vscode.Uri.parse('data:text/plain,' + encodeURIComponent(bundledTemplate))` as specified in the PRD Technical Considerations section.

  ```
  ├── plugin.json
  ```

## Warnings (should fix)

- [ ] **F001: Stale-index status-bar warning not implemented** — PRD UI/UX section

  The PRD specifies: "If the index is stale (> 24 h since last `buildIndex` and `cohereApiKey` is set), a status-bar item shows a warning icon '⚠ Semantic index stale — run Update'. Clicking it runs `legion.update`." This check-on-stale behaviour is part of AC-1.3.2's implied maintenance contract. No status-bar registration for this warning was found in `findEntity.ts` or `extension.ts`.

  **Remediation:** Add a check after `buildIndex` returns (or on `findEntity` invocation) that compares `cache.lastIndexed` against `Date.now() - 86_400_000`. Show a status-bar warning item if stale and `cohereApiKey` is set.

- [ ] **F006: PR comment missing "ADRs Filed" table** — `src/commands/installPrReviewBot.ts:131-245`

  AC-6.4.5 requires: "An 'ADRs Filed' table (if any) lists: ADR title, link to the ADR file on the PR branch." The `getWorkflowTemplate()` function generates the PR comment body without an ADRs Filed section. The current comment body only has Pages Created, Pages Updated, and Contradictions Detected sections, matching AC-6.4.2–6.4.4 but omitting 6.4.5.

- [ ] **F007: CLAUDE.md routing block references `decisions/` instead of `adrs/`** — `src/context/claudeMdWriter.ts:69`

  The `buildLegionBlock()` function generates this routing instruction:

  ```typescript
  // claudeMdWriter.ts:69
  4. Check \`${wikiPath}/decisions/\` for architecture decisions affecting the relevant domain.
  ```

  The PRD Layer 1 technical design specifies the path as `adrs/`:

  **Remediation:** Change `decisions/` to `adrs/` on `claudeMdWriter.ts:69`.

## Suggestions (consider improving)

- [ ] **F002: Server SDK variant deviates from PRD pattern** — `src/mcp/legionMcpServer.ts:13-18`

  The PRD Technical Considerations section shows `McpServer` with the `server.tool()` convenience method for registering tools with Zod schemas. The implementation instead uses the lower-level `Server` class with manual `setRequestHandler(ListToolsRequestSchema, ...)` and `setRequestHandler(CallToolRequestSchema, ...)` handlers. Both approaches are valid with `@modelcontextprotocol/sdk`, and the implemented version offers more control (e.g., input validation and routing in a single switch). No functional gap exists, but the implementation diverges from the documented pattern, which could confuse developers consulting the PRD.

---

- [ ] **F003: `bundled/export-templates/` files specified in PRD are inlined** — `src/driver/wikiExport.ts`

  The PRD "Files Touched" section lists three new bundled template files:
  - `bundled/export-templates/docusaurus.config.js.hbs`
  - `bundled/export-templates/html-shell.html.hbs`
  - `bundled/export-templates/html-style.css`

  All three are instead inlined into `wikiExport.ts` as string constants/functions (`generateDocusaurusConfig()`, `htmlShell()`, `HTML_STYLE_CSS`). This reduces the file count and avoids a bundling dependency (no Handlebars needed), which is a reasonable implementation decision. The CSS especially benefits from being inline since it's performance-critical. However, it means users cannot customise the HTML shell without editing compiled extension code.

---

- [ ] **F005: `resolveRepoRoot` return type and resolution order diverge from PRD** — `src/util/repoRoot.ts:19-77`

  The PRD declares `resolveRepoRoot()` as returning `Promise<string | string[]>` (to support the "All roots" array case). The implementation returns `Promise<string>`, with `"__all__"` as a sentinel value for the "All roots" selection. Callers must special-case `"__all__"` rather than receiving an array. Additionally, the PRD resolution order places "Persisted session selection" at step 2 (before "single root fast-path"), but the implementation checks single root at step 2 and session state at step 3 — which is more efficient but diverges from the documented contract.

  Neither divergence is a functional regression, but both could surprise future maintainers relying on the PRD as the specification.

---

- [ ] **F008: Obsidian plugin resides in monorepo subfolder instead of separate repo** — `companion-plugins/legion-obsidian/`

  The PRD Phase 1 step 1 specifies: "Create new repo `legion-obsidian` from Obsidian sample plugin template." The implementation places the plugin under `companion-plugins/legion-obsidian/` inside the main `vibecode-legion` repo. While all source files are present (`LegionStatusPanel.ts`, `ContradictionInboxView.ts`, `AnnotateButton.ts`, `triggerUpdate.ts`, `dependencyGraph.ts`, `configReader.ts`, `main.ts`), shipping it in the main repo means the Obsidian community registry submission process (Phase 4) must reference a subdirectory, and paired versioning (`obsidian-plugin@1.0.0` ↔ `legion@1.x`) is harder to communicate. No functional gap, but the distribution strategy and CI setup will need adjustment if the separate-repo plan is pursued.
